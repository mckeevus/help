Stage 1 · Before you open Claude
Get Clear on What You Want
The most common mistake is jumping straight into building before knowing what you actually want to see. Spend 20 minutes on this first — it saves hours later.
1✓
Write down your 3–5 most important reporting questions
Not charts. Not metrics. Questions. The dashboard exists to answer questions you currently have to dig for.

Examples:
  • Where are we vs budget this month, by cost centre?
  • Which categories are running over and by how much?
  • What does our spend trend look like over the last 6 months?
  • Which departments are the biggest variance drivers?
Write yours down before moving on. If you can't articulate the question, the chart won't be useful.
2✓
Build a rough mockup to get the visual shape right
You don't need to know how to build it — just sketch what it should look like. Two good options:

Option A — Lovable (easiest): Go to lovable.dev, describe what you want in plain English and it'll generate a visual prototype. Free to try.

Option B — Claude: Go to claude.ai, describe your dashboard idea and ask it to generate a mockup as an HTML file you can open in your browser.
This is just a reference, not the final product. You're figuring out layout — how many panels, what charts, what tables — so you have something to point at when you start building for real.
3✓
Decide how you want to update it
This is a critical decision that shapes the whole build. Pick one:

  • Drop a CSV and it refreshes — you export from NetSuite/your finance system, drop the file in, dashboard updates. Simplest. Recommended to start.
  • Paste data in manually — you copy numbers into a spreadsheet or form that feeds the dashboard. Works but more manual.
  • Live connection to NetSuite — dashboard pulls data automatically via API. Most powerful but most complex. Save this for later once you've proven the concept.
Recommendation: start with CSV drop. It's fast to build, easy to understand, and you can upgrade to a live connection later once you know exactly what you need.
Stage 2 · Before you open Claude
Understand Your Data
You need to know what data you have, how it's structured, and what it can actually tell you before Claude can help you build anything meaningful.
1✓
Export a sample from NetSuite and look at it
Pull a real export — even just last month — and open it in Excel. Look at it as raw data, not a report.

Ask yourself:
  • What are the column names exactly? (Not what you call them — what the system calls them)
  • What format are the dates in? (DD/MM/YYYY, YYYY-MM-DD, text?)
  • Are dollar amounts formatted as numbers or text with $ signs?
  • Are there any blank rows, header rows, or footer totals that would confuse a program?
  • Is there a unique identifier per row — a transaction ID, invoice number, or similar?
The boring stuff matters. A date formatted as text instead of a date will break charts. A $ sign in a number column will break calculations. Find these now, not later.
2✓
Map what each reporting question needs
Take each question from Stage 1 and identify which columns answer it.

Example:
Example mapping
Question: "Where are we vs budget by cost centre?"
→ Needs: cost_centre, actual_amount, budget_amount, period

Question: "Which categories are running over?"
→ Needs: category, actual_amount, budget_amount
If a column you need doesn't exist in the export, that's important to find out now. You may need to add it to the export template in NetSuite, or combine two columns.
3✓
Understand what's defined where
Some things are calculated in the data itself. Some are defined by business rules that live in someone's head. Know which is which.

  • Is budget loaded into NetSuite, or does it live in a spreadsheet somewhere?
  • Are cost centres and department codes consistent month to month, or do they change?
  • What counts as "over budget"? Is there a threshold, or is any variance flagged?
  • Are there categories or codes that should be excluded from certain reports?
Write these rules down. You'll need to give them to Claude explicitly — it can only work with what you tell it.
Stage 3 · Before you open Claude
Get a Clean Data Export
Before designing anything, get the actual data out of NetSuite and clean it up. A good dashboard built on messy data is still a mess.
1✓
Export at least 3 months of data as CSV
In NetSuite, find the report or saved search that most closely matches what you need and export it as CSV. Get at least 3 months so you can test trending and comparisons.

If NetSuite isn't the source of truth for everything — e.g. budget lives in Excel — export both. You'll need to tell Claude where each piece of data comes from.
2✓
Clean it up in Excel before using it
Fix the things you found in Stage 2. Common ones:

  • Remove header rows and footer totals — the data should start on row 1
  • Make sure column names have no spaces or special characters (use underscores: cost_centre not Cost Centre)
  • Make sure dates are formatted consistently
  • Remove $ signs and commas from number columns so they're pure numbers
  • Delete any completely blank rows
Save a "raw" copy and a "clean" copy. Never edit the raw export — always work from a copy.
3✓
Know your numbers before you build
Before building the dashboard, manually calculate a few key numbers from your data — total spend for a month, biggest over-budget category, etc.

Why this matters: When Claude builds the dashboard, you need to be able to verify it's correct. If the dashboard shows $847,000 for October and your spreadsheet says $851,000, you'll know something's wrong. If you don't know the right answer going in, you can't catch errors.
Stage 4 · In Claude
Define the Scope with Claude
Don't ask Claude to build anything yet. First, work with it to define exactly what you're building, agree on the MVP, and develop the master prompt together. This is the most important stage.
1✓
Open Claude and describe what you want in plain English
Go to claude.ai and start a new conversation. Don't ask it to build anything — just describe the project. Use this as a starting point:

Starter prompt — copy and edit this
I want to build a financial reporting dashboard for my business. I'm the financial controller and I want to replace manual reporting with something visual and interactive.

My data comes from: NetSuite (actuals) and Excel (budget)
I want to be able to answer these questions:
1. [Your question 1]
2. [Your question 2]
3. [Your question 3]

My data is updated: [monthly/weekly/daily]
I want to load data by: dropping in a CSV file each month

I don't have a coding background. Can you help me understand what's realistic to build, what the simplest version would look like, and what decisions I need to make before we start building?
Let Claude ask you questions. A good response will include clarifying questions — answer them honestly. This conversation is doing the architectural thinking so the build goes smoothly.
2✓
Upload your sample CSV and ask Claude to analyse it
Once you've described the project, upload your cleaned CSV. Then ask:

Follow-up prompt
Here's a sample of my actual data. Can you look at the structure and tell me:
1. Whether it has everything needed to answer my reporting questions
2. Any problems with the data format you can see
3. What columns we'll use for what
Claude will spot things you missed. Date format issues, inconsistent category names, missing columns — it'll flag them before they become problems in the build.
3✓
Agree on the MVP — what's in, what's out
Before building, explicitly agree with Claude on what version 1 includes. Ask it:

Scoping prompt
Based on everything we've discussed, what would a solid MVP look like? Tell me exactly what it would include, what it wouldn't include yet, and what we'd add in a version 2. I want to make sure we're aligned before you start building anything.
Don't skip this step. Getting agreement on scope before building is what stops you ending up with something half-finished or that tries to do too much at once.
4✓
Get Claude to write the master prompt
Once you're aligned on scope, ask Claude to produce a single master prompt that captures everything — the data structure, the requirements, the rules, the MVP. You'll use this as the brief for the actual build.

Master prompt request
Great. Can you now write a master prompt that summarises everything we've agreed — the data structure, the reporting requirements, the business rules, the MVP scope, and any technical decisions we've made? I'll use this as the brief when we start building.
Save this master prompt. It's your north star for the whole build — and if something goes wrong later, you can start a fresh conversation and paste it in to get back on track instantly.
Stage 5 · In Claude
Build, Test, and Iterate
Now you build. The work you did in the previous stages means Claude has everything it needs. Your job now is to test, validate, and add on once the core works.
1✓
Start the build with your master prompt
Start a new Claude conversation, paste in your master prompt, then add:

Build kickoff
Using this brief, please build the MVP dashboard. Start with the data loading and the first two charts only. I want to verify those are working correctly before we add anything else.
Build in small pieces. Ask for one section at a time and test each one before moving on. It's much easier to fix a small piece than unpick a fully built dashboard.
2✓
Validate the numbers before you trust it
Once Claude gives you a working dashboard, load your real data and check the numbers against what you calculated manually in Stage 3.

  • Does total spend for a known month match?
  • Does the biggest over-budget category match your spreadsheet?
  • Do trends look right directionally — peaks and dips where you'd expect them?
If something doesn't match, tell Claude exactly what you expected vs what you got. It'll diagnose and fix it. Never assume it's right — always verify with numbers you already know.
3✓
Add things one at a time once the core works
Once the base is validated, add your additional requirements one by one. For each new feature:

  • Describe what you want in plain English
  • Ask Claude to add it without breaking what's already working
  • Test it with real data before moving to the next feature
Resist the urge to ask for everything at once. One change at a time means you always know what broke something if something breaks.
4✓
Understand how to maintain it yourself
Before you consider the dashboard "done", ask Claude:

Maintenance prompt
Can you explain to me in plain English: how do I update this with new data each month? And if something breaks — like a column name changes in my export — how do I fix it? I want to be able to maintain this myself without coming back to you every time.
You own this tool. Claude built it with you, but it's yours. Most maintenance tasks — updating data, tweaking a chart, fixing a column name — are a 5-minute conversation with Claude, not a developer job.

Key principles to keep in mind

Start narrow. One data source, 3 charts, one clear question answered well. Expand from there.
Data quality first. A beautiful dashboard on dirty data gives you beautiful wrong answers. Fix the data before you build.
Know the right answer before you test. Always validate with numbers you already trust.
Don't login to everything via GitHub. Use an email address. Keep credentials separate per service.
You own it. This isn't something a developer built for you — it's something you built with Claude. That means you can fix it, change it, and extend it yourself.
